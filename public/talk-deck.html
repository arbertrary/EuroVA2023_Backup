What defines Provenance?

“[…] the sources of information, such as entities and processes, involved in producing or delivering an artifact”

World Wide Web Consortium (W3C) (Missier, Belhajjame, and Cheney 2013)

“[…] provenance is information about entities, activities, and people involved in producing a piece of data or thing, which can be used to form assessments about its quality, reliability or trustworthiness”

W3C (Groth and Moreau 2013)

What defines Provenance?

->

The World Wide Web consortium defines provenance as “the sources of information involved in producing an artifact”

->

and that this information can be used to assess the artifacts “quality, reliability or trustworthiness”

This mentions the issue of quality assessment. For this, let’s take another quick step back and take a look at how to possibly assess the value of a visualization

The value of visualization

vanWijk (2005): How to assert and quantify the value of a visualization.

The value is a trade-off between costs and the gained knowledge.

“Just claiming that a visualization gives insight is not enough, if we want to offer additional value.”

Dou, Ribarsky, and Chang (2010): Capture a user’s reasoning process by capturing internal (i.e. in-app interaction) and external (e.g. think-aloud, manual notes) information

These captured reasoning processes and the visualization state history are what we in this case understand as Provenance

The value of visualization

In 2005, van Wijk proposed a model on how to assert and quantify the value of a visualization.

The model considers the costs of creating a visualization, and the costs of making sense of it. This is compared to the gained insight and increase in knowledge.

-> But you need to somehow be able to “prove” that the visualization gives insight.

This is where provenance information comes into play.

-> Dou et al. for example extend van wijks model by proposing ways how to capture a user’s reasoning process in visual analysis workflows.

With this information you then have proof or at least arguments for the value of your visualization.

-> These reasoning processes together with the state history of the visualization application are what we in this case understand as provenance

The “Digital Lab Book”

The Digital Lab Book has grown from two critical problems:

a) How to do field work in geosciences when you cannot physically get to the “field”

b) The lack of reproducibility and transparency in geoscience (immersive) visual analytics workflows

The two main tasks that form the basis of the Digital Lab Book were how to support field work if you cannot get to the field.

(Remember the notebook drawing on the geological field work slide.)

And the lack of provenance in geoscientific visual analysis processes.

What DLB is …

  • Third-party tool
  • Capturing of visualization states
  • Branching and “What-if?” scenarios
  • Supports explicit externalization of thought processes
  • Export and serialization of visual analysis provenance

… and what it is not.

  • An elaborate undo/redo button
  • Low-level event-driven interaction tracking (mouse clicks etc.)

What DLB is and what it is not

The Digital Lab Book is a tool acting as third party to a given visualization application.

It captures visualization states at user-defined time steps and enables returning to previous states, branching and therefore the exploration of “What-if?” scenarios in your analysis workflow.

It supports taking notes and creating mind maps which means explicit externalization of reasoning processes.

It also allows to export and serialize the gathered artifacts of visual analysis provenance which can then be published.

->

It is, however, not just an elaborate undo/redo button and history management software.

And it does not track low-level events and interactions such as mouse clicks but rather distinct states.

Implementation Choices



  • Web-based application
  • Angular frontend and Python Flask backend
  • Git as provenance “database”

Implementation Choices

We chose to develop the DLB as a web-based application to keep the possibility of potentially offering its functionality as Software as a service.

The frontend is written in Angular and the backend is a simple Flask architecture.

For the provenance management and as a “database” we use python bindings to libgit2 since Git versioning offers much that we desired for such a tool.

And this also allows easy sharing and therefore publication for example by submitting the git repository folder as supplement or by uploading the repository to github or gitlab

Interaction with Visualization Applications

Minimum requirements for a visualization application to be potentially supported:

  • API (websocket, REST API) or scriptable/plugin-api
  • Save/export current project
  • Load/import a project during runtime

Interaction with Visualization Applications

Of course the DLB can’t simply connect to any and all visualization applications. The currently supported applications all had to be slightly modified in some way.

The minimum requirements are a sort of API that allows to save the current state of the application and visualization as a file and load such a file again during runtime to return to the exact state as before.

In the best case this is a file that can be version controlled and for which a file diff can be calculated

-> now let’s take a look at the DLB interface

Dashboard

Dashboard

The dashboard is the “landing page” and leads a user through a multi-step process (with skippable steps) to setup the repository and connection to the visualizatioon application.

It also allows to export and import a packaged provenance archive which in simple terms is a git repository containing all visualization states, thumbnails, metadata for the repository

History View

History View

The history view combines a provenance or history graph with a scrollable list view of the latest n states of the currently checked out branch.

There’s also a replay feature which is able to send back a sequence of states to the visualization application.

Clicking on a thumbnail of the scene opens a pop-up window of the state with more detailed information.

You can return to a state, create a branch from that state, add a note, or add the state to the mind map

Mind Map

Mind Map

The mind map is a currently rather simple implementation for knowledge organization and includes basic features like creating and linking nodes, creating clusters from multiple nodes, color picking etc

Future Work

Development

  • Support immersive workflows (e.g. motion tracking in ARENA2)
  • Support more visualization applications
  • Polishing (never finished)

Evaluation

  • Qualitative evaluation in selected in-depth use cases with domain scientists
  • Quantitative evaluation in user studies with domain scientists and non-specialists (e.g. University students)

Future Work

There is always something more to do obviously.

The most important points for the Digital Lab Book now are to support immersive analytics workflows and track provenance artifacts that arise there. We have an optitrack setup in the ARENA2

Then what’s more important from a research standpoint is the evaluation. We plan to do this by focussing both on a selection of in-depth use cases of domain experts.

But also with a somewhat larger user study

Conclusion

The Digital Lab Book is a third-party provenance capturing and management tool with the original goal of supporting visual analysis use cases at our ocean research institute.
